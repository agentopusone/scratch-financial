# Resilient Scraper Architecture

## 1. Introduction
This scraper is built to operate inside an intentionally hostile environment. The target page behaves like a real-world integration surface that does not want to be scraped. It randomizes class names, rotates between multiple layouts, delays rendering, and occasionally simulates rate limits. The scraper’s purpose is not just to extract data, but to demonstrate how a resilient integration client behaves when the upstream system is unstable, unpredictable, or adversarial. The design emphasizes adaptability, observability, and graceful degradation rather than brittle correctness.

## 2. Handling Structural Drift
The most common form of hostility is structural drift. The page changes its class names, removes IDs, and reorganizes the DOM without warning. The scraper never relies on a single selector. Instead, every extraction function uses a layered strategy. It begins with the most direct path, using selectors that work when the page is stable. When those fail, it falls back to structural heuristics, such as identifying any table with enough rows to plausibly represent invoice data or locating text that resembles a client label. If the DOM is too inconsistent to navigate structurally, the scraper drops to a text-based approach, scanning the raw innerText of the page and recovering values through pattern matching. This layered approach ensures that even when the page drifts, the scraper continues to extract meaningful data.

## 3. Handling Layout Variants
The hostile site rotates between multiple layout variants. Instead of guessing which structure is active, the scraper begins every run by detecting the layout. It checks for the presence of structural markers that uniquely identify each variant. Once the layout is known, the scraper logs it and adapts its extraction strategy accordingly. This early detection step prevents wasted work and provides a clear audit trail when the upstream system changes its behavior. Layout detection is intentionally lightweight, but it gives the scraper a way to orient itself before attempting extraction.

## 4. Handling Timing and Rendering Delays
Another form of hostility comes from unpredictable rendering. The page may delay the appearance of key elements, load content asynchronously, or render parts of the invoice only after user-like interactions. The scraper never assumes the DOM is ready when the page finishes loading. It waits for specific anchors inside the iframe and wraps all extraction operations in a retry mechanism. The retry wrapper uses exponential backoff with jitter, which makes the scraper behave more like a human and prevents it from hammering the page. Every retry attempt is logged with the reason for failure and the delay before the next attempt. This approach allows the scraper to survive slow rendering, partial loads, and transient failures without collapsing.

## 5. Handling Selector Breakage
Selectors are the most fragile part of any scraper. The hostile site randomizes class names and occasionally removes attributes entirely. To survive this, the scraper treats selectors as hints rather than guarantees. When a selector fails, the scraper does not treat it as a fatal error. It falls back to structural heuristics and text-based extraction. This mirrors the way a human would extract information from a broken page. If the labels are missing, look at the table. If the table is broken, read the text. If the text is messy, infer the values from context. The scraper’s design reflects this human-like adaptability.

## 6. Handling Rate Limits and Transient Failures
The hostile site occasionally simulates rate limits or transient failures. The retry wrapper absorbs these without interrupting the scrape. When an extraction attempt fails due to a missing element or a temporary error, the scraper waits, logs the event, and tries again. The exponential backoff ensures that repeated failures do not overwhelm the page. This pattern is common in real-world integrations, where upstream systems may throttle requests or behave inconsistently under load.

## 7. Handling Inconsistent or Messy Data
Even when the scraper successfully extracts values, the raw data may be inconsistent. Currency symbols, whitespace, and formatting vary across layout variants. The scraper includes a normalization layer that converts these raw values into a consistent schema. Totals are parsed into numbers, currency symbols are stripped, and fields like Subtotal, Tax, and Total are mapped into canonical names. This normalization step ensures that downstream systems do not need to understand the quirks of the hostile page. It also makes the scraper’s output predictable even when the input is not.

## 8. Handling Observability and Debuggability
A scraper that operates in hostile conditions must be observable. Every meaningful event, including layout detection, retries, extraction results, and normalization output, is logged as a structured JSON object. This creates a complete narrative of the scrape, making it easy to diagnose drift, identify failing selectors, or understand why a fallback strategy was triggered. The logs are designed to be machine-readable but human-interpretable, which is essential for debugging integrations that fail intermittently.

## 9. Handling Worst-Case Scenarios
The scraper is designed to degrade gracefully. If the page is intact, it takes the fast path. If the page is partially broken, it falls back to heuristics. If the page is severely degraded, it falls back to text scanning. And if all else fails, the architecture already has a natural place to insert an LLM-based repair loop, where the scraper can hand the HTML to a model and request structured JSON. Even without that final layer, the scraper demonstrates the core principles of adversarial integration engineering: resilience, adaptability, and a bias toward returning useful data even under imperfect conditions.
