# Human Interest Case Study: Integrating With a Legacy PEO

## Context
When Human Interest evaluated mid‑market prospects, the integration team lead was always brought in before a contract was drafted. The sales team needed to know whether a customer’s payroll or PEO system could support the level of automation required for daily payroll ingestion. In this particular case, the prospect insisted on keeping their existing PEO. The PEO was a legacy, multi‑employer system with no API, no automated export mechanism, and a web portal designed entirely for human navigation. The question was not whether we could pull data once. The question was whether we could build a reliable, unattended pipeline that ran every day without a human in the loop.

## Discovery
During the pre‑sales review, I was given a demo login, the customer’s payroll calendar, and the list of fields we needed to ingest. It became clear within the first hour that the PEO offered no programmatic surface at all. Every piece of payroll data lived behind a sequence of screens that required a person to click through menus, select a pay period, open a roster, and then drill into individual employee records. The structure of these screens changed depending on the pay period, the number of employees, and even the time of day. Some reports included headers and totals, others did not. Some tables loaded immediately, others required a manual refresh. Session timeouts were inconsistent and often occurred mid‑navigation. None of this was malicious. It was simply a system that had never been designed with automation in mind.

## Payroll Ingestion Constraints
Because Human Interest needed a daily ingestion pipeline, we could not rely on manual exports or customer intervention. The integration had to run on its own, every morning, with predictable timing and high reliability. That requirement shaped the entire design. We needed a way to navigate the PEO’s workflows exactly as a human would, extract data from pages that did not have stable identifiers, and handle the small but constant changes that occurred in the layout of the portal.

## Design
The design discussions focused on whether we could automate the PEO’s UI in a way that was stable enough for payroll deadlines. We walked through the full workflow and mapped out every screen the automation would need to visit. We identified the points where the PEO changed structure from week to week and the places where the system occasionally returned incomplete data. We also looked at the timing patterns of the portal, since some pages loaded slowly and others required a second navigation step before the data appeared. The conclusion was that the only viable approach was to automate the browser directly and build an extraction layer that could tolerate drift in the HTML. We already had some parts of this automation build for other clients, with basic scraping.

## Adversarial Approach
I built a Playwright‑based automation client that reproduced the exact sequence of actions a payroll administrator would take. The client logged in, navigated through the menus, selected the correct pay period, opened the roster, and extracted the data from the rendered tables. Because the HTML structure was inconsistent, the extraction logic relied on text proximity and semantic cues rather than fixed selectors. When the PEO changed the layout of a table or renamed a field, the extraction layer could still locate the correct values. For cases where the structure changed more significantly, the system captured the HTML snippet and passed it to a human interpret the fields and map them back to our canonical schema. We did not use LLM because payroll contains PII, and 401K needs strict SEC compliance. There is no guidance on LLMs yet.

## Ingestiion Pipeline
To support daily ingestion, the automation was wrapped in a reliability layer that handled retries, session refresh, idempotent writes, and structured logging. If the PEO timed out or returned an incomplete page, the system retried the step rather than failing the entire run. If the PEO changed the structure of a report, the drift detection logic flagged the change and fell back to the semantic extraction path. The goal was to ensure that payroll ingestion completed every day without requiring a human to intervene or babysit the process.

## Result
The result was a fully automated, daily ingestion pipeline built on top of a system that had no automation surface at all. The integration ran reliably, supported the customer’s payroll schedule, and allowed Human Interest to onboard a client that competitors could not support. More importantly, it established a repeatable pattern for integrating with legacy payroll and PEO systems that were never designed for third‑party access.

This is the same class of adversarial integration Scratch faces with PIMS and other legacy veterinary systems. It is not about defeating sophisticated defenses. It is about building stable, resilient automation on top of inconsistent, human‑oriented workflows and delivering a reliable pipeline where none exists.
