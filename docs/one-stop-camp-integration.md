# OneStopCamp Case Study: Integrating With Wix and WordPress Summer Camp Provider Sites

## Context
At OneStopCamp, we auto-onboarded after‑school and camp providers, in all of Seattle, who used a wide range of website builders and registration tools. Many of the smaller operators did not use dedicated scheduling or enrollment software. Instead, their entire program catalog lived on a Wix or WordPress site that had been assembled over several years by different staff members. The structure of these sites varied dramatically from provider to provider. Some pages listed programs in long scrolling sections. Others used blog posts as program entries. Some embedded Google Sheets or PDFs. Others mixed HTML tables, images, and free‑form text. There was no consistent schema, no API, and no export mechanism. Yet we needed to ingest their program data reliably so families could search, filter, and enroll through OneStopCamp.

## Integration Challenges
The integration process always began with a discovery pass. When a provider signed up, we reviewed their website to understand how their programs were represented. In many cases, the site looked simple to a human reader but was structurally chaotic underneath. A single program might have its name in a header tag, its age range in a paragraph, its dates inside an image caption, and its price inside a styled span element. Some providers updated their sites weekly, others only once a season. The challenge was not to scrape a page once. The challenge was to build a system that could extract structured program data from a site that had no structure at all and do it repeatedly as the provider updated their content.

## Design
The design discussions focused on how to build a scraper that could survive this level of inconsistency. We walked through several real provider sites and mapped out the patterns we saw. Wix sites tended to generate deeply nested divs with autogenerated class names that changed whenever the provider edited the page. WordPress sites varied depending on the theme and plugins in use. Some used custom post types. Others used page builders that produced HTML that looked different every time the provider made a small change. We needed an approach that did not rely on fixed selectors or brittle assumptions about layout.

## Implementation
I built a scraper that treated each provider site as a semi‑structured document rather than a predictable HTML tree. The scraper fetched the page, parsed the HTML, and then applied a series of semantic extraction passes. It looked for text patterns that indicated program names, dates, times, age ranges, and prices. It analyzed the proximity of these elements to one another to determine which pieces of information belonged together. When the structure was too inconsistent for deterministic rules, the scraper captured the relevant HTML snippet and passed it through an LLM to interpret the content and map it back to our canonical program schema. This allowed us to handle cases where a provider embedded a PDF, used an image with text baked in, or wrote their program descriptions in long paragraphs.

Because providers updated their sites frequently, the scraper needed to be resilient to drift. I added logic that compared the current extraction results to previous runs and flagged anomalies. If a provider changed the layout of their program list, the scraper adapted by falling back to the semantic extraction path. If a field disappeared or changed format, the system logged the change and continued processing rather than failing outright. The goal was to ensure that OneStopCamp always had up‑to‑date program information without requiring the provider to change how they managed their website.

## Ingestion Pipeline
To support daily ingestion, the scraper ran on a schedule and wrote its results through an idempotent pipeline. If a provider updated only one program, the system detected the change and updated that record without duplicating or overwriting unrelated data. If the provider removed a program from their site, the system marked it as inactive. This allowed OneStopCamp to present accurate, real‑time program listings even though the underlying data came from websites that were never designed to be data sources.

## Result
The result was a reliable ingestion pipeline that allowed OneStopCamp to integrate with providers who had no modern software at all. We were able to onboard camps and after‑school programs that competitors could not support because we could extract structured data from whatever surface the provider happened to use. This experience maps directly to the kind of adversarial integration Scratch requires. It is the same pattern of taking a human‑oriented, inconsistent, semi‑structured system and building a stable, automated pipeline on top of it.
